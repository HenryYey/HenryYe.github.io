我们都知道 Promise.all可以并发执行异步任务。但它的最大问题就是如果其中某个任务出现异常(reject)，所有任务都会挂掉，直接进入reject 状态。

模块规范化

这里只讲CommonJs和ESMoudle。

CommonJS模块是对象，是运行时加载，运行时才把模块挂载在exports之上（加载整个模块的所有），加载模块其实就是查找对象属性。
ES Module不是对象，是使用export显示指定输出，再通过import输入。此法为编译时加载，编译时遇到import就会生成一个只读引用。等到运行时就会根据此引用去被加载的模块取值。所以不会加载模块所有方法，仅取所需。

简单来说，CommonJs是

module.exports ={ sth }
exports.sth

const{ sth } = require('sth')

而 ESModule是

exportconst a
import{ a } from'path'

exportdefault a
export * as a from'path'

CommonJS和ES Module是可以写一起的，但是最好不要。毕竟一个是编译阶段一个是运行阶段。免得遇到一些坑。

async & await

详解见这里

const Func = asyncfunction(userName, password) {

理解和用法都很简单，也是常用的，但是不是每个函数都可以用，一些比较原始的例如node的mysql模块，它对数据库的操作就不能被await作用，所以这时候我们就要用下面这个通用方法了。

浅谈react中setState的同步和异步机制

发表于 2019-04-11 更新于 2020-06-27 分类于 react

场景引入

相信大家在使用React的时候都会遇到关于setState()的问题，例如以下场景:

本意是改变输入框时，对应替换state.text。但是如果在setState() 后打印一下就会发现，state的改变总是‘慢一拍’

比如你输入123时，控制台会先后打印 ‘ ‘ ,’1’, ‘12’ 。 经过思考和翻看文档，不难发现，这是因为setState()是异步的。如果换一种场景，在输入123后点击按钮，获取输入框的内容:

打印出来的是’123’而不是’12’。这也证明了state的改变只是’慢一拍’而不是’少一拍’。那么React为什么要引用这个异步机制呢？

#原理原理


原理可以用这张图来描述，即在react中，setState通过一个队列机制实现state的更新。当执行setState时，会把需要更新的state合并后放入状态队列，而不会立刻更新this.state，当进入组件可更新状态时，这个队列机制就会高效的批量的更新state。

保证内部的一致性

目前 React 提供的 objects 有三种：① state ② props ③ refs
这三个 object 的行为和表现都是一致的，它们能够保证构建出协调的树。举个栗子，假设state是同步更新的，子组件的state更新了，但是父组件又需要该state，就会进行状态提升。而props并不会立即更新，因为组件并没有重新渲染。那么state、props更新一次，组件就会重新渲染一次。这样会显著的降低性能。所以在React中，props和state的更新都是异步的。

###提升性能

这就是关于上面提到的性能问题。再举几个例子:
当你和别人聊天时，你正在打字，对方突然发了很多条消息过来，如果state同步更新的话，对话框组件就会一直重新渲染，此时你的输入框就会卡一下。如果state是异步的话，react会进行批量更新，’偷偷’渲染好组件，虽然会有一丢丢延迟，但是体验感就好多了。
再例如，一个网站的设计是切换页面时弹出loading。如果你的网速很快，很快就能获取另一个页面，如果state是同步更新的，那么loading还是会被渲染，在你的屏幕里一闪而过再切换到页面。那么异步的时候，react会先渲染好页面，如果渲染地很快，那么加载动画不会弹出。此时就能做到无缝切换了。对于计算机而言，这样提升了性能，对于用户而言，提升了体验感。

如何进行同步更新

面试常问的一个点: 如何让setState()同步更新？
根据官方文档， 在setState()里给予其一个回调函数，在状态更新完毕后进行调用，这个过程就是同步的。下面我来测试一下:

[object Object]

大家在console.log的时候肯定见到过[object Object]这种输出，那么这是什么呢?
先说一下产生原因把

const res = {
  name: 'Tony',
  id: 123
}
console.log(`res${res}`) // res [object Object]
// 相当于
console.log(res.toString()) // res [object Object]

形如这样的输出会造成这样的结果，其实类似于强制转化，因为前面已经有字符串了，再+一个变量就会输出这个变量的字符串形式，
也就是toString()，这个函数返回 “[object type]”，其中type是对象的类型，如果改成console.log(‘res’,res)就不会有问题

##利用方括号访问对象属性

var obj = {

    age: 20// 字段会自动打上双引号 "age"

var a = "age"
console.log(obj["name"]) // henryye
console.log(obj[a]) // 20

内存泄漏

稍微深入了解js的朋友们肯定知道浏览器的垃圾回收机制，而防止因为未销毁的全局变量造成的内存泄漏的方法就是在函数尾部将该变量设为null，
但是这种做法很憨而且很麻烦。一般情况下，码农们并没有刻意去设置null或者是规范的代码的防止了内存泄露，而性能的影响也不是那么大（除非老板追求极致的性能或者你是强迫症患者）。像react这种组件化框架可以说有效的防止了内存泄漏，除非你在该组件卸载时又创建了变量。所以，平时代码写规范点就ok啦~

输出顺序如代码所示，本质上是考对事件队列的宏任务和微任务的理解，推荐一篇博文

作用域

for (var i = 0; i < 5; i++) {
    setTimeout(function() {
console.log(i);
    }, 1000);
}

// 5 5 5 5 5

因为 setTimeout 的 console.log(i); 的i是 var 定义的，所以是函数级的作用域，不属于 for 循环体，属于 global。等到 for 循环结束，i 已经等于 5 了，这个时候再执行 setTimeout 的五个回调函数，里面的 console.log(i); 的 i 去向上找作用域，只能找到 global下 的 i，即 5。所以输出都是 5。

如果要输出12345，则需要改变函数的作用域

一：

for (var i = 0; i < 5; i++) { 
    (function(i){      //立刻执行函数
        setTimeout(function (){
console.log(i);  
         },1000);  
    })(i);  
}

二:

for (let i = 0; i < 5; i++) {     //let 代替 var
    setTimeout(function (){
console.log(i);  
     },1000);  
}

sort()方法

熟悉js的童鞋都知道sort() 方法用于对数组的元素进行排序(快排)，并返回数组，但是很少人钻研过它的细节，以至于大部分人都踩过坑。别急，首先来康康定义：默认排序顺序是根据字符串UniCode码。因为排序是按照字符串UniCode码的顺序进行排序的，所以首先应该把数组元素都转化成字符串（如有必要），以便进行比较。

语法：arrayObject.sort(sortby);

参数sortby 可选，用来规定排序的顺序，但必须是函数。

// 普通用法
var arr = ['henry','ye','tonyma']
arr.sort() // ["henry", "tonyma", "ye"]

// 踩坑了
var arr = [123,321,1000]
arr.sort() // [1000, 123, 321]

这时候就要给sort()提供一个函数参数了，这个函数的规则如下：
比较函数应该具有两个参数 a 和 b，其返回值如下：

若 a 小于 b，即 a - b 小于零，则返回一个小于零的值，数组将按照升序排列。

若 a 等于 b，则返回 0。

若 a 大于 b, 即 a - b 大于零，则返回一个大于零的值，数组将按照降序排列。
形如

var arr = [123,321,1000]
arr.sort((a,b)=> {
return a-b;
}) // [123, 321, 1000]

这个函数还可以通过ifelse返回不同的值形成更复杂的形式呢，万变不离其宗，记住a和b的关系和返回值与升降序的关联就好啦
